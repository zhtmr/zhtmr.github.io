# post-structuring — 초안을 포스트로 구조화

사용자 초안/메모를 이 블로그에 바로 올릴 수 있는 잘 구조화된 본문으로 다듬는다. **의미를 바꾸지 않고 구조·포맷·표현만** 손본다. 블로그 전체 컨벤션은 `../blog-pipeline/references/blog-conventions.md`를 함께 참조한다.

## 왜 구조화하는가
독자(개발자)는 글을 위에서 아래로 정독하지 않고 **제목과 코드블록으로 스캔**한다. 한 덩어리 텍스트는 기술 글에서 가독성을 무너뜨린다. 그래서 논리 단위마다 제목을 부여하고, 코드·명령·로그는 펜스로 분리해 시각적 닻을 만든다. 이 원리를 이해하면 어떤 초안이 와도 어디를 끊을지 판단할 수 있다.

## 절대 규칙: 의미 불변
- 사실·수치·주장·인과관계·코드 동작을 바꾸지 않는다.
- 내용을 **추가하거나 삭제하지 않는다.** 초안에 없는 설명을 지어내지 않는다. 빠진 게 있어 보이면 본문에 임의로 채우지 말고 `<!-- STRUCTURE-NOTES -->`에 "사용자 확인 필요"로 남긴다.
- 코드를 "고치지" 않는다. 코드가 틀려 보여도 그 판단은 tech-reviewer의 몫이다. 여기서는 포맷팅(들여쓰기 정렬, 언어 태그, raw 래핑)만 한다.

## 구조화 체크리스트
1. **제목 계층**: 글 전체 제목은 frontmatter `title`이 담당하므로 본문 최상위 섹션은 `##`로 시작한다. 하위는 `###`. 한 단계씩만 내려간다(`##` 다음 바로 `####` 금지).
2. **도입 인용 블록**: 글 첫머리에 그 포스트가 다루는 내용을 한두 문장으로 요약한 `> 인용 블록`을 둔다(기존 포스트의 관습). 초안에 요약 문장이 있으면 그것을 쓰고, 없으면 만들지 말고 NOTES에 제안만 남긴다.
3. **코드블록**: 모든 코드/명령/로그를 ` ```언어 ` 펜스로 감싼다. 언어 태그 필수(`java`, `yml`, `bash`, `sql`, `json` 등 — `blog-conventions.md`의 언어 매핑 참조). 들여쓰기는 원본 의미를 유지하며 일관되게 정렬.
4. **Liquid 충돌 방지**: 코드/텍스트에 `{{ }}` 또는 `{% %}`가 그대로 있으면 해당 블록을 `{% raw %}` … `{% endraw %}`로 감싼다. 안 감싸면 Jekyll 빌드가 깨지거나 내용이 사라진다 — 이것이 과거 빌드 오류의 흔한 원인이다.
5. **이미지**: 이미지 참조는 외부 호스팅 컨벤션(`blog-conventions.md` 참조)을 따른다. 사용자가 로컬 경로/파일명만 줬으면 임의로 URL을 완성하지 말고 NOTES에 "이미지 경로 확정 필요"로 남긴다. 정렬이 필요하면 `{: .align-center}` 같은 기존 클래스를 쓴다.
6. **리스트/표**: 나열은 불릿/번호 리스트로, 비교·매핑은 표로. 단, 원문에 없던 항목을 채워 넣지 않는다.
7. **TOC**: 기본 `toc: false`. 섹션이 많아 목차가 유용하면 frontmatter에 `toc: true` 제안을 NOTES에 남긴다(확정은 publisher/사용자). 특정 제목을 목차에서 빼려면 그 제목 바로 아래 줄에 `{:.no_toc}`.

## 입출력
- 입력: `_workspace/00_input.md`. 재호출 시 `_workspace/02_review_*.json` + `_workspace/03_decisions.md`.
- 출력(초기): `_workspace/01_structured.md` = 구조화 본문 + 잠정 frontmatter(아는 값만 채우고 모르는 건 비움) + 끝에 `<!-- STRUCTURE-NOTES -->` 주석으로 (a)적용한 변경 (b)확인 필요 모호점.
- 출력(재호출): `_workspace/04_final.md` = 확정 수정 반영본 + 적용/보류 findings 기록.

## 수정 반영(재호출) 모드
검수 리포트의 **확정된** findings만 적용한다(병렬 편집 충돌 방지 — 미확정 findings는 손대지 않는다). 각 finding의 `suggested_fix`/`correction`을 해당 위치에 반영하되, 적용 시 의미가 바뀔 위험이 있으면 적용하지 말고 NOTES에 "사용자 판단 필요"로 남긴다. 어떤 finding을 적용/보류했는지 `<!-- STRUCTURE-NOTES -->`에 id 단위로 기록한다.
