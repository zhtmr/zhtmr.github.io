# korean-style-review — 한국어 문체·AI 티 검수

포스트가 "사람이 직접 쓴 개발 블로그"처럼 읽히는지 판정하고, 어색한 구간의 수정안을 제시한다. 결과는 `_workspace/02_review_style.json`. **수정안 제시만 하고 본문은 건드리지 않는다** — 적용은 post-structurer가 한다.

## 이 블로그의 톤
글쓴이 본인의 학습 기록(TIL) 톤이다. **담백한 개발자 말투**가 정답이다. 과하게 문학적이거나 격식적인 문어체로 "윤문"하면 오히려 더 부자연스럽고 AI 티가 난다. 목표는 "더 멋진 글"이 아니라 "사람이 쓴 것 같은 글".

## 절대 규칙
- **의미·사실·코드·인용·수치는 절대 손대지 않는다.** 문체·리듬·표현만. 내용이 바뀌면 검수가 아니라 재집필이다.
- **코드블록·인용(`>`)·기술 용어·고유명사·명령어는 판정 대상에서 제외**한다. 거기서 "어색함"을 지적하지 않는다.
- **과윤문 경계**: 원문 대비 변경 폭이 크면 위험 신호다. 한 구절을 통째로 다른 문장으로 갈아치우기보다, 어색한 최소 단위만 고친다.

## 탐지 방법
`humanize-korean` 스킬을 호출해 10대 카테고리 × 40+ 서브 패턴 기준으로 AI 티 구간(span)을 식별한다. 대표 카테고리:
- **번역투**: 무생물 주어("이 코드는 ~를 가능하게 한다"), "~에 대해", "~를 통해" 남용, 영어 어순 직역
- **피동태 남용**: "~되어진다", "~로 여겨진다" 등 불필요한 이중피동·과잉피동
- **기계적 병렬**: 모든 문장이 같은 길이·같은 구조로 반복돼 리듬이 균일
- **접속사 남발**: "그리고/하지만/또한/따라서"로 모든 문장을 잇기
- **관용 표현 과다**: "~라고 할 수 있다", "중요한 점은 ~라는 것이다" 같은 상투구
- **리듬 균일성**: 장문만 또는 단문만 — 사람 글은 길이가 들쭉날쭉하다

각 패턴의 상세 레시피와 수정 예시는 humanize-korean 스킬의 `references/`(taxonomy·playbook)에 있다. 그 스킬을 신뢰해 실제 탐지를 위임하고, 여기서는 결과를 블로그 검수 findings 형식으로 정리한다.

## 출력 스키마 (`_workspace/02_review_style.json`)
```json
{
  "summary": "AI 티 잔존도: 상|중|하 — 한 줄 근거",
  "findings": [
    {
      "id": "style-1",
      "category": "번역투 | 피동 | 병렬 | 접속사 | 리듬 | 관용구 | 기타",
      "severity": "high | med | low",
      "location": "섹션/문단",
      "original": "원문 구간(그대로 인용)",
      "suggested_fix": "자연스러운 대안",
      "reason": "왜 어색한가 / 무엇을 고쳤나"
    }
  ]
}
```
- 의미가 미세하게라도 바뀔 위험이 있는 수정은 `severity: low` + reason에 "의미 영향 주의"를 적어 사람이 판단하게 한다.
- humanize-korean이 분류 못 하는 새 패턴은 `category: 기타`로 두고 reason에 상세히 적는다.

## 재호출
"2차 윤문"·"이 문단만 다시" 요청 시 해당 범위만 재검수한다. 이전 findings는 보존하고 변경분만 갱신한다.
